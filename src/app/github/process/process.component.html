What: GitHub is a web-based platform that provides version control and collaboration features for software development using Git.

Why: GitHub is essential because it enables teams to collaborate, track changes, manage codebases, and automate workflows through pull requests, issues, and CI/CD integrations.

Where: GitHub is used across open-source projects, private enterprise applications, DevOps pipelines, documentation repositories, and anywhere software collaboration happens.

Initial Repository Setup

Creating Repository on GitHub

GitHub Web Interface Steps:

  1. Go to github.com and sign in
  2. Click "New" button (green button) or go to github.com/new
  3. Fill repository details:
    • Repository name: my-awesome-project
    • Description: Brief project description
    • Public/Private: Choose visibility
    • ✅ Add README file
    • ✅ Add .gitignore (choose your language template)
    • ✅ Choose license (MIT recommended for open source)
  4. Click "Create repository"

Clone Repository to Local Machine

1

Get Repository URL and Clone

# Navigate to your projects directory
cd ~/Desktop/projects

# Clone using HTTPS (recommended for beginners)
git clone https://github.com/yourusername/my-awesome-project.git

# Or clone using SSH (if you have SSH key setup)
git clone git@github.com:yourusername/my-awesome-project.git

# Enter the project directory
cd my-awesome-project

# Verify remote connection
git remote -v
Success! Repository cloned successfully. You should see "origin" pointing to your GitHub repository.
2

Set Up Initial Project Structure

# Create project folders
mkdir src docs tests assets

# Create initial files
touch src/index.js src/app.js
touch docs/README.md docs/API.md
touch tests/test.js
touch .env.example

# Check what was created
ls -la

# Add all files to git
git add .

# Make first commit
git commit -m "feat: initial project structure and files"

# Push to GitHub
git push origin main

Adding and Managing Team Members

Adding Collaborators

Adding Team Members Steps:

  1. Go to your repository on GitHub
  2. Click "Settings" tab (at the top of repo)
  3. Click "Collaborators" in left sidebar
  4. Click "Add people" button
  5. Enter teammate's GitHub username or email
  6. Select permission level and click "Add [username]"
  7. Team member will receive email invitation
Permission Level Can Do Cannot Do Best For
Read View, clone, download code Push changes, create PRs from forks only Stakeholders, viewers
Write Push code, create branches, PRs, issues Change settings, delete repo Developers, contributors
Admin Everything including settings, delete Nothing restricted Team leads, owners

Set Up Branch Protection Rules

Branch Protection Steps:

  1. Go to repository → Settings → Branches
  2. Click "Add rule" under "Branch protection rules"
  3. Enter branch name pattern: main
  4. Configure these essential settings:
Require pull request reviews before merging - Forces code review
Require status checks to pass - Ensures CI/CD passes
Require branches to be up to date - Prevents merge conflicts
Include administrators - Rules apply to everyone
Restrict pushes that create files larger than 100MB
Important: Once you enable branch protection, you CANNOT push directly to main branch. You must use Pull Requests.

How to Create Merge Request (Pull Request)

Creating Feature Branch and Making Changes

1

Create Feature Branch

# Make sure you're on main branch
git checkout main

# Pull latest changes
git pull origin main

# Create and switch to new feature branch
git checkout -b feature/user-authentication

# Verify you're on the new branch
git branch
2

Make Changes and Commit

# Make your code changes in your editor
# Example: Edit src/auth.js, add login function

# Check what files changed
git status

# Add specific files
git add src/auth.js

# Or add all changes
git add .

# Commit with descriptive message
git commit -m "feat: add user login authentication system"

# Push branch to GitHub
git push -u origin feature/user-authentication

Creating Pull Request on GitHub

Pull Request Steps:

  1. Go to your repository on GitHub
  2. You'll see yellow banner: "Compare & pull request" - click it
  3. Or click "Pull requests" tab → "New pull request"
  4. Select branches:
    • Base: main (where you want to merge TO)
    • Compare: feature/user-authentication (your branch)
  5. Fill PR details:
    • Title: Clear, descriptive title
    • Description: What changes you made and why
    • Reviewers: Select team members to review
    • Labels: Add relevant labels (bug, feature, etc.)
  6. Click "Create pull request"
Pro Tip: Create a Pull Request template by adding .github/pull_request_template.md to your repository for consistent PR descriptions.

Review Process

Code Review Steps

Leave feedback - Comment on specific lines if needed
Approve or Request Changes

# Test the code, run tests

Local Merge Process

2

Merging Pull Requests Locally

Fetch PR branch - Get the latest changes from remote
Switch to main - Ensure you're on the target branch
Perform merge - Use appropriate merge strategy
Push changes - Update remote repository 
# Method 1: Direct merge (preserves commit history)
git checkout main
git pull origin main
git merge feature/user-authentication
git push origin main

# Method 2: Squash merge (single clean commit)
git checkout main
git pull origin main
git merge --squash feature/user-authentication
git commit -m "Add user authentication feature"
git push origin main

# Method 3: Cherry-pick specific commits
git checkout main
git cherry-pick commit-hash-1 commit-hash-2
git push origin main

GitHub Web Interface Merge

3

Merging via GitHub UI

Navigate to PR - Go to Pull requests tab in repository
Verify status - Ensure all checks pass and reviews approve
Click merge button - Choose merge strategy from dropdown
Edit commit message - Customize merge commit details
Delete branch - Clean up feature branch after merge
GitHub Merge Requirements: Make sure branch protection rules are satisfied, all required status checks pass, and you have appropriate permissions to merge.

Merge Options

Merge Commit

Creates merge commit, preserves branch history

Use when: You want to preserve complete history
Squash and Merge

Combines all commits into one clean commit

Recommended: Keeps main branch history clean
Rebase and Merge

Replays commits without merge commit

Use carefully: Can rewrite commit history

Merge Conflicts Resolution

Understanding Merge Conflicts

Conflict occurs when: Two people edit the same line of code in different branches, and Git can't automatically decide which version to keep.
1

When Conflict Happens

# You'll see this error when trying to merge
CONFLICT (content): Merge conflict in src/app.js
Automatic merge failed; fix conflicts and then commit the result.

# Check which files have conflicts
git status

# You'll see files under "Unmerged paths"
2

Resolving Conflicts Manually

Open the conflicted file in your editor. You'll see conflict markers:

<<<<<<< HEAD
// Your current branch's version
function loginUser() {{ '{' }} return 'User logged in'; {{ '}' }}
=======
// Incoming changes from other branch
function loginUser() {{ '{' }} return 'Authenticated user'; {{ '}' }}
>>>>>>> feature/user-authentication
Resolution Steps:
  1. Choose which version to keep (or combine both)
  2. Remove conflict markers (<<<<<<<, =======, >>>>>>>)
  3. Test your code to ensure it works
  4. Add the resolved file: git add filename
  5. Commit the resolution: git commit -m "resolve: merge conflict in filename"
3

Complete Merge Process

# After resolving all conflicts and testing
git add .
git commit -m "resolve: merge conflicts between main and feature/user-authentication"
git push origin feature/user-authentication

# Then complete the Pull Request on GitHub

After Merge Cleanup

# Switch back to main branch
git checkout main

# Pull latest changes (includes your merged PR)
git pull origin main

# Delete local feature branch
git branch -d feature/user-authentication

# Delete remote branch (GitHub usually does this automatically)
git push origin --delete feature/user-authentication
